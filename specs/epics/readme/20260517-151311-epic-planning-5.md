# Epic EP-005: Distribution and Documentation

Status: review

## Goal
`scip-search` users and maintainers can install, build, verify, and document the Go CLI through supported release and source workflows.

## Context
The README already presents `scip-search` as a thin Go CLI with installation paths for latest release installs, specific versions, branch builds, custom install directories, local clone installs, and `--version` verification. This epic bounds the distribution-facing behavior around those workflows so query epics can remain focused on command behavior and SCIP traversal.

## Personas
- **Automation Agent**: an AI or script-driven caller running terminal commands in constrained environments, needing reproducible install commands and a quick way to verify the installed binary.
- **CLI Maintainer**: a Go developer preparing releases for macOS and Linux users, needing packaging checks and README examples that match the supported install behavior.
- **Source Builder**: a developer or early adopter with Go and make available, installing from a branch or local clone before a release artifact exists.

Applies to: the entire epic scope.

### Completion Criteria
Story Writers can produce implementation stories where users can install a released `scip-search` binary, build and install from source workflows, confirm the installed build with `scip-search --version`, and maintain README plus release validation coverage that proves supported distribution commands remain accurate.

### Non-Functional Requirements
- NFR-000-1: Distribution workflows must stay separate from SCIP indexer installation; users are told which indexers they need, but `scip-search` install does not install language indexers.
- NFR-000-2: Install and source-build workflows must support automation: commands have deterministic success criteria, actionable failure output, and no dependency on an interactive UI.
- NFR-000-3: Packaging tests and release validation must not require large real-world repositories or external language indexer execution.
- NFR-000-4: README installation examples must remain aligned with the supported install script options, local build command, and version verification behavior.

### Related External Components
- Component C-001 - Release installer: The user-facing `install.sh` workflow invoked from the README for latest release, specific version, branch, and custom directory installs.
- Component C-002 - Release artifacts: The published binaries or archives consumed by release install workflows.
- Component C-003 - Local Go and make toolchain: The caller-provided tools required for branch and local clone source builds.
- Component C-004 - README installation documentation: The user-facing installation and verification guidance maintained with the distribution behavior.

### Interfaces
- I-001-001 - Installer invocation contract (Interface 001 of Component C-001): The shell invocation and environment options users rely on for release, branch, and custom-directory installs.
- I-003-001 - Source build contract (Interface 001 of Component C-003): The local clone build and install command surface used by source builders.
- I-004-001 - Version verification contract (Interface 001 of Component C-004): The command-line version output users and release automation use to confirm the installed executable.

### Out of Scope
- Query command behavior for `symbols`, `packages`, `references`, and `implementations`.
- Shared query runtime semantics for `--index`, JSON stdout, stderr diagnostics, and process exit statuses.
- SCIP protobuf parsing, traversal views, fixtures for query data, or query-specific golden JSON cases.
- Installing language indexers, generating SCIP indexes, daemon/watch mode, incremental updates, custom index formats, ctags fallback behavior, MCP server behavior, UI, graph visualization, embeddings, vector search, and semantic similarity.
- Release hosting provider setup, CI provider selection, signing/notarization policy, package manager formulas, container images, and operating systems beyond the macOS/Linux install scope stated in the README.
- Implementation code; this is a planning artifact for downstream Story Writer tasks.

### Assumptions
- **ASM-000-1**: The quick install command installs the latest released `scip-search` binary by default, while `VERSION` selects a specific release - *Why*: The README labels the command "latest release" and gives `VERSION=v1.0.0` as the specific-version option - Confidence: HIGH.
- **ASM-000-2**: `BRANCH` means a source build workflow that requires Go and make rather than a released binary lookup - *Why*: The README says "Build from a branch (requires Go and make)" - Confidence: HIGH.
- **ASM-000-3**: macOS and Linux are the supported installer platforms for this epic; architecture selection details can be specified by story acceptance criteria - *Why*: The README scopes quick install to macOS/Linux but does not list CPU architectures - Confidence: MEDIUM.

## Capability CAP-001 - Install released scip-search binaries

`scip-search` users can install a requested released version on macOS or Linux into a chosen binary directory and verify the installed executable without needing a local clone.

### References
- goal spec: `README.md#installation`, lines 131-148 - Defines quick release install, `VERSION`, and `INSTALL_DIR` examples.
- goal spec: `README.md#installation`, lines 157-160 - Defines `scip-search --version` as the verification command.
- task: `epic-planning-5` scope - Includes install script expectations, user-facing docs impact, and release-facing validation.

### Description
This capability defines the release install path for users who want a ready-to-run binary. It covers selecting the default latest release, selecting an explicit version, installing into a caller-selected directory, and producing enough observable outcome for users or automation to know whether the executable is available. It excludes branch and local clone source builds, which are handled by a separate capability.

## Capability CAP-003 - Report CLI version information

`scip-search` users and automation can run `scip-search --version` and receive version information that identifies the installed build well enough to confirm installation and release provenance.

- goal spec: `README.md#installation`, lines 157-160 - Defines `scip-search --version` as the verification command.

### Description
This capability defines the user-facing version check that confirms an installed or source-built executable is the intended `scip-search` binary. It covers version output for released builds and source builds enough for automation and maintainers to distinguish what was installed. It does not redefine query command output, shared JSON stdout behavior, or query runtime error conventions.
