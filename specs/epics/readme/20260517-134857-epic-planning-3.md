# Epic EP-003: Symbol and Package Discovery Queries

Status: review

## Goal
`scip-search` users can discover full SCIP symbols from partial symbol names and enumerate indexed packages with deterministic query-specific JSON responses.

## Context
`scip-search` is a thin one-shot CLI over a caller-selected SCIP index. This epic covers the query behavior for the two discovery commands: resolving `symbols --name` input into full SCIP symbol strings that can feed later symbol-based commands, and listing packages with optional prefix filtering so agents can inspect what an index contains before choosing a symbol query.

## Personas
- **Automation Agent**: an AI or script-driven caller running `scip-search` in a terminal or sandbox, needing deterministic JSON results and exact full SCIP symbol strings for follow-up commands.
- **CLI Maintainer**: a Go developer maintaining `scip-search`, needing bounded query semantics and fixtures that keep symbol and package behavior stable across SCIP indexes.
- **Story Writer**: a downstream planning agent that needs enough product behavior to write implementation stories without re-reading the source README.

### Completion Criteria
Story Writers can produce implementation stories where `symbols --name` returns deterministic partial-name matches including exact full SCIP symbol strings, `packages` returns deterministic package identities with optional literal prefix filtering, both commands define query-specific JSON payloads under the shared runtime contract, and fixtures prove matching, filtering, ambiguous matches, empty results, and stable ordering.

### Non-Functional Requirements
- NFR-000-1: Query stories must preserve full SCIP symbol strings exactly as stored in the index so callers can pass them unchanged to later `references` or `implementations` commands.
- NFR-000-2: Query stories must not redefine shared runtime behavior owned by `epic-planning-1` or raw SCIP traversal behavior owned by `epic-planning-2`.
- NFR-000-3: Successful query payloads must be stable for automation: deterministic ordering, no diagnostics in stdout, and explicit empty result collections for successful no-match cases.
- NFR-000-4: Query-specific fixtures must be deterministic, small enough for normal CLI test runs, and valid through the same SCIP loading and traversal path used by the commands.

### Interfaces

- I-001-002 - Package discovery query contract (Interface 002 of Component C-001): The `packages` query returns package identities from the index, optionally filtered by package-name prefix.

- **ASM-000-1**: Successful no-match discovery queries return empty result collections rather than shared runtime errors - *Why*: A valid query with no indexed match is different from invalid invocation or index-loading failure, and automation needs a machine-readable successful absence case - Confidence: HIGH.

## Capability CAP-001 - Resolve partial symbol names

`scip-search` users can query `symbols --name <name>` with a partial symbol name and receive deterministic matches that include the full SCIP symbol strings needed by later symbol-based commands.

### References
- goal spec: `README.md#scip-symbol-format`, lines 37-52 - Requires partial name queries such as `--name Supervisor` to resolve to full SCIP symbols and return them alongside results.
- goal spec: `README.md#what-is-scip-search`, lines 67-80 - Documents `scip-search symbols --index <index-path> --name <name>`.
- goal spec: `README.md#complementary-existing-tool`, lines 114-120 - Frames `symbols` as the query used to find where a named symbol is defined.
- sibling boundary: `epic-planning-2` output CAP-001 and CAP-002 - provide symbol inventories, document paths, and source-location metadata consumed by symbol query stories.

### Description
This capability defines how callers discover symbols when they know only a simple or partial name. The query matches the supplied `--name` value against symbol name or descriptor text, returns every matching definition candidate rather than choosing one silently, and treats ambiguous partial names as a successful multi-result response. The result payload exposes a `symbols` collection where each item includes the exact full `symbol` string, the matched display or descriptor name, package identity, and definition location when traversal provides one.

### Out of Scope
- Exact-symbol lookup for `references` or `implementations` commands.
- Reference occurrence selection, implementation relationship traversal, or relationship expansion from a matched symbol.

- Shared invocation failures, index-loading failures, stdout/stderr rules, and process exit behavior.

### Assumptions
- **ASM-001-1**: If multiple symbols share the same simple name or partial-name substring, returning all matches is preferable to failing as ambiguous - *Why*: the README says partial queries resolve to full symbols and the caller can choose the returned full symbol for later commands - Confidence: HIGH.
- **ASM-001-2**: Stable ordering can be specified by observable result values rather than by traversal iteration order - *Why*: automation callers need deterministic JSON, while the source spec does not require preserving SCIP document order for discovery output - Confidence: MEDIUM.

## Capability CAP-002 - List indexed packages with prefix filtering

`scip-search` users can run `packages` with or without `--prefix` and receive deterministic package identities for the indexed packages that match the requested prefix.

### References
- goal spec: `README.md#scip-symbol-format`, lines 37-43 - Defines the package-related components of SCIP symbol strings.
- goal spec: `README.md#what-is-scip-search`, lines 67-80 - Documents `scip-search packages --index <index-path> [--prefix <prefix>]`.
- goal spec: `README.md#complementary-existing-tool`, lines 114-120 - Frames `packages` as the query used to answer what packages exist.
- sibling boundary: `epic-planning-2` output CAP-001 - provides the symbol and external symbol inventory from which package identities can be exposed to query planners.

### Description
This capability defines package discovery from the caller-selected SCIP index. Without `--prefix`, the command returns all indexed package identities that can be derived from SCIP symbol prefixes; with `--prefix`, it returns only packages whose package name begins with the literal prefix. The result payload exposes a `packages` collection where each item identifies the scheme, package manager, package name, package version, and an exact package key suitable for display or comparison by automation.

### Out of Scope
- Installing indexers, generating package indexes, querying package registries, dependency graph traversal, module graph analysis, or version resolution.
- Filtering by descriptor text, symbol name, document path, language, package manager, package version, or regular expression.
- Reference, implementation, or symbol definition result schemas.
- Shared invocation failures, index-loading failures, stdout/stderr rules, and process exit behavior.

### Assumptions
- **ASM-002-1**: `--prefix` applies to the package-name component of the SCIP symbol format, not to the scheme, package manager, version, or descriptors - *Why*: the command is named `packages`, and the task scope says package listing and prefix filtering rather than generic symbol-prefix filtering - Confidence: MEDIUM.
- **ASM-002-2**: Package results are de-duplicated by full package identity before prefix filtering output is finalized - *Why*: many symbols can belong to the same package, while the command answers what packages exist - Confidence: HIGH.

## Capability CAP-003 - Validate discovery queries with fixtures

`scip-search` maintainers can validate symbol and package query stories against deterministic SCIP fixtures and golden JSON cases that cover matching, filtering, empty results, and ambiguous names.

### References
- goal spec: `README.md#scip-symbol-format`, lines 37-52 - Provides example full SCIP symbols and the partial-name-to-full-symbol behavior fixtures must exercise.
- goal spec: `README.md#what-is-scip-search`, lines 67-80 - Lists the symbol and package command forms that fixture-backed tests must cover.
- sibling boundary: `epic-planning-2` output CAP-004 - owns shared traversal fixtures for SCIP data coverage, while this capability owns query-specific cases and expected command payloads.
- task: `epic-planning-3` scope - explicitly includes tests and fixtures for symbol and package query classes.

### Description
This capability defines the query-specific validation surface for symbol and package discovery. Fixtures cover at least one exact-looking partial match, one partial substring match, multiple symbols with overlapping names, packages that share and do not share a prefix, and successful empty result cases. Golden JSON cases prove that full SCIP symbol strings, package identities, ordering, and empty collections stay stable without depending on external indexer installation.
