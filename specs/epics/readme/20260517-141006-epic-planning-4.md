# Epic EP-004: Reference and Implementation Queries

Status: review

## Goal
`scip-search` users can query exact SCIP symbols for references and implementations and receive deterministic JSON results with source locations, ranges, missing-symbol outcomes, and fixture-backed coverage.

## Context
`scip-search` is a thin one-shot CLI over a caller-selected SCIP index. This epic covers the two symbol-based query commands that use full SCIP symbols returned by discovery: `references --symbol` and `implementations --symbol`. It bounds exact-symbol matching, reference occurrence selection, implementation relationship traversal, command-specific result payloads, and query fixtures while relying on sibling epics for shared CLI runtime behavior and raw SCIP traversal views.

## Personas
- **Automation Agent**: an AI or script-driven caller running `scip-search` in a terminal or sandbox, needing exact full-symbol queries, stable JSON, and source ranges it can feed into later code-navigation or editing workflows.
- **CLI Maintainer**: a Go developer maintaining `scip-search`, needing bounded reference and implementation semantics that stay aligned with SCIP occurrence roles and relationship fields.
- **Story Writer**: a downstream planning agent that needs enough product behavior to write implementation stories for symbol-based queries without re-reading the README, SCIP schema, or sibling epics.

### Completion Criteria
Story Writers can produce implementation stories where `references --symbol` and `implementations --symbol` accept exact full SCIP symbols, return deterministic successful JSON payloads with source document paths and SCIP ranges, handle missing exact symbols without shared runtime failure, and validate reference occurrence and implementation relationship behavior with deterministic fixtures and golden JSON cases.

### Non-Functional Requirements
- NFR-000-1: Query stories must preserve exact full SCIP symbol strings as caller-provided or traversal-provided values so automation can chain `symbols`, `references`, and `implementations` commands without rewriting identifiers.
- NFR-000-2: Successful query payloads must be stable for automation: deterministic ordering, explicit empty result collections for successful no-match cases, and no diagnostics in stdout.
- NFR-000-3: Query stories must preserve SCIP source-location semantics, including document-relative paths and schema-defined 0-based ranges, without reading source files to infer missing locations.
- NFR-000-4: Query stories must not redefine shared CLI runtime behavior, raw SCIP parsing, traversal view construction, symbol discovery, package discovery, index generation, daemon behavior, or custom index formats.
- NFR-000-5: Query-specific fixtures must be deterministic, small enough for normal CLI test runs, and valid through the same SCIP loading and traversal path used by the commands.

### Related External Components
- Component C-001 - SCIP symbol identifiers: Exact full symbol strings supplied through `--symbol` and returned in query result payloads.
- Component C-002 - SCIP occurrence data: Document-relative source ranges and role bitsets used to distinguish reference occurrences from definitions.
- Component C-003 - SCIP relationship data: Symbol relationship edges used to include related reference targets and implementation results.
- Component C-004 - Query fixture set: Deterministic SCIP test data and expected JSON cases used by maintainers to validate references and implementations.

### Interfaces
- I-001-001 - Reference query contract (Interface 001 of Component C-001): The `references --symbol` query accepts an exact full SCIP symbol and returns matching reference locations.
- I-001-002 - Implementation query contract (Interface 002 of Component C-001): The `implementations --symbol` query accepts an exact full SCIP symbol and returns implementation symbol locations derived from SCIP implementation relationships.

### Assumptions
- **ASM-000-1**: Successful exact-symbol queries with no matching symbol or related results return an empty result collection instead of a shared runtime error - *Why*: missing indexed matches are valid query outcomes, while sibling runtime errors are reserved for invalid invocation and index-loading failures - Confidence: MEDIUM.
- **ASM-000-2**: `references --symbol` treats definition occurrences as symbol context, not reference results - *Why*: the SCIP schema defines occurrences without `SymbolRole.Definition` as references, and the README frames references as call/use sites - Confidence: MEDIUM.
- **ASM-000-3**: `implementations --symbol` returns symbols connected by SCIP implementation relationships and their available definition locations, not arbitrary type-hierarchy or text matches - *Why*: SCIP exposes `Relationship.is_implementation`, while the README excludes semantic similarity and custom analysis - Confidence: HIGH.

## Capability CAP-001 - Resolve exact symbol query inputs

`scip-search` users can pass an exact full SCIP symbol to `references --symbol` or `implementations --symbol` and receive a deterministic successful response for found and missing symbols without partial-name resolution.

### References
- goal spec: `README.md#scip-symbol-format`, lines 37-52 - Full SCIP symbol strings returned by discovery are used by later references and implementations calls.
- goal spec: `README.md#what-is-scip-search`, lines 67-80 - Documents `scip-search references --index <index-path> --symbol <scip-symbol>` and `scip-search implementations --index <index-path> --symbol <scip-symbol>`.
- sibling boundary: `specs/epics/readme/20260517-134857-epic-planning-3.md#capability-cap-001---resolve-partial-symbol-names` - Owns partial symbol discovery before callers choose an exact symbol.

### Description
This capability defines the common symbol input behavior for both symbol-based commands. The commands interpret `--symbol` as an exact full SCIP symbol string, do not perform partial-name matching, and preserve the queried symbol in the successful payload. A syntactically valid exact symbol that is absent from the traversal view is a successful no-match result with an empty command-specific collection, while invalid invocation shape remains owned by the shared runtime epic.

## Capability CAP-002 - Return reference occurrences for exact symbols

`scip-search` users can run `references --symbol <scip-symbol>` and receive deterministic reference occurrence results for the exact symbol and SCIP reference-related symbols, including document paths and ranges.

### References
- goal spec: `README.md#what-is-scip-search`, lines 67-80 - Documents the references command form.
- goal spec: `README.md#complementary-existing-tool`, lines 114-120 - Frames references as the query used to answer what calls or uses a symbol.
- official source: `https://raw.githubusercontent.com/scip-code/scip/main/scip.proto` - Defines occurrence ranges, `SymbolRole.Definition`, and `Relationship.is_reference`.
- sibling boundary: `specs/epics/readme/20260517-100328-epic-planning-2.md#capability-cap-003---expose-occurrence-and-relationship-lookup-views` - Provides occurrence and relationship lookup views for reference planners.

### Description
This capability defines the successful `references` query result behavior. The query returns source occurrences whose symbol matches the exact queried symbol or a SCIP reference-related symbol and whose role is not a definition. Each result includes enough stable context for automation to locate the occurrence: the matched occurrence symbol, document path, SCIP range, role context, and the originally queried symbol in the response envelope.

## Capability CAP-003 - Return implementation relationship results

`scip-search` users can run `implementations --symbol <scip-symbol>` and receive deterministic implementation results derived from SCIP implementation relationships, including implementation symbols, definition locations, and ranges when available.

### References
- goal spec: `README.md#what-is-scip-search`, lines 67-80 - Documents the implementations command form.
- goal spec: `README.md#complementary-existing-tool`, lines 114-120 - Frames implementations as the query used to answer what implements a symbol.
- official source: `https://raw.githubusercontent.com/scip-code/scip/main/scip.proto` - Defines `Relationship.is_implementation` and symbol relationships.
- sibling boundary: `specs/epics/readme/20260517-100328-epic-planning-2.md#capability-cap-003---expose-occurrence-and-relationship-lookup-views` - Provides relationship and occurrence lookup views for implementation planners.

### Description
This capability defines the successful `implementations` query result behavior. The query returns symbols connected to the queried exact symbol by SCIP implementation relationships, with definition locations included when traversal data provides them. The payload distinguishes the queried symbol, each implementation symbol, the relationship basis, and any available document path and SCIP range so automation can navigate to concrete implementations without deriving type hierarchy itself.

## Capability CAP-004 - Validate symbol-based queries with fixtures

`scip-search` maintainers can validate reference and implementation query stories against deterministic SCIP fixtures and golden JSON cases that cover exact symbols, missing symbols, reference occurrences, implementation relationships, locations, ranges, and stable ordering.

### References
- goal spec: `README.md#what-is-scip-search`, lines 67-80 - Lists the reference and implementation command forms that fixture-backed tests must cover.
- goal spec: `README.md#scip-symbol-format`, lines 37-52 - Provides example full SCIP symbols and chaining from discovery to symbol-based commands.

- task: `epic-planning-4` scope - Explicitly includes tests and fixtures for references and implementations query classes.

### Description
This capability defines the query-specific validation surface for symbol-based commands. Fixtures cover at least one exact symbol with direct reference occurrences, one missing exact symbol, one reference-related symbol relationship, one implementation relationship with a definition location, one implementation relationship without a location if traversal can represent it, and successful empty results. Golden JSON cases prove that queried symbols, matched symbols, document paths, ranges, ordering, and empty collections stay stable without depending on external indexer installation.
