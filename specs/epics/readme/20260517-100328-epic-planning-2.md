# Epic EP-002: SCIP Data Traversal

Status: review

## Goal
`scip-search` query planners can consume official SCIP index data through a bounded traversal layer that preserves the documents, symbols, occurrences, relationships, source ranges, and hover metadata needed by the documented CLI queries.

## Context
`scip-search` is a thin Go CLI that answers code-intelligence questions from pre-built SCIP indexes without compiling, type-checking, watching, or maintaining a daemon. This epic covers the common traversal foundation under query-specific epics: it defines what SCIP data must be loaded, indexed in memory, and exposed so symbol, package, reference, and implementation query stories can focus on query behavior instead of parsing the SCIP protobuf structure themselves.

## Personas
- **Automation Agent**: an AI or script-driven caller running `scip-search` in a terminal or sandbox, expecting deterministic query results from an explicit SCIP index path.
- **CLI Maintainer**: a Go developer maintaining `scip-search`, needing a small traversal contract that stays aligned with the official SCIP schema and does not invent a custom index format.
- **Story Writer**: a downstream planning agent that needs enough SCIP traversal context to write implementation stories for query commands without re-reading the source README or SCIP schema.

### Completion Criteria
Story Writers can produce implementation stories for a SCIP traversal layer that loads official SCIP data, exposes reusable in-memory lookup views for query planners, preserves source range and hover metadata, and defines fixtures that prove the traversal layer supports the documented query classes without implementing query-specific result behavior.

### Non-Functional Requirements
- NFR-000-1: Traversal stories must use the official Go SCIP bindings and schema fields as the source of truth for parsing and interpretation.
- NFR-000-2: Traversal stories must not introduce a custom persisted index format, daemon, watch mode, incremental update mechanism, embedding store, vector store, UI, MCP server, or ctags fallback behavior.
- NFR-000-3: Traversal output consumed by query planners must preserve enough source-location detail to let query-specific epics decide final JSON schemas without re-parsing SCIP data.
- NFR-000-4: Fixture stories must be deterministic and small enough to validate traversal behavior in normal CLI test runs.

### Related External Components
- Component C-001 - Official SCIP Go bindings: Go package used to read SCIP protobuf messages and access schema-defined document, symbol, occurrence, relationship, range, and hover fields.
- Component C-002 - SCIP protobuf schema: protocol definition that defines semantic meaning for index, document, symbol, relationship, occurrence, range, symbol role, and position encoding data.

### Interfaces
- I-001-001 - SCIP traversal input (Interface 001 of Component C-001): The traversal layer receives the loaded SCIP data boundary owned by the CLI runtime epic and interprets it through official binding types only.
- I-001-002 - Query planner traversal view (Interface 002 of Component C-001): Query-specific stories receive reusable in-memory views over documents, symbols, occurrences, relationships, ranges, and hover metadata without owning SCIP parsing.

### Out of Scope
- CLI command structure, shared flags, index-path validation, stdout/stderr conventions, exit status semantics, and shared runtime errors.
- `symbols --name`, `packages --prefix`, `references --symbol`, and `implementations --symbol` matching semantics or result JSON schemas.
- Index generation, installation of language indexers, file watching, incremental refresh, daemon behavior, caching policy beyond one process, and cross-run persistence.
- Custom index formats, ctags fallback languages, embeddings, semantic similarity, vector storage, graph visualization, UI, and MCP server behavior.
- Implementation code; this is a planning artifact for downstream Story Writer tasks.

### Assumptions
- **ASM-000-1**: Query planners need one reusable traversal view rather than each command walking raw SCIP documents independently — *Why*: the planned commands all depend on the same SCIP documents, symbols, occurrences, relationships, and ranges, while sibling epics own command-specific semantics — Confidence: HIGH.
- **ASM-000-2**: Traversal fixtures may use compact synthetic SCIP indexes when they preserve schema-valid documents, occurrences, symbols, relationships, ranges, and hover metadata — *Why*: the README requires reading SCIP output directly but does not require fixtures to be produced only by external indexers — Confidence: MEDIUM.

## Capability CAP-001 - Load SCIP index data for query traversal

`scip-search` query planners can receive a traversal-ready view of the caller-selected SCIP index through the official Go SCIP bindings without owning protobuf parsing.

### References
- goal spec: `README.md#language-support` — official SCIP bindings are required for parsing and traversal.
- official source: `https://pkg.go.dev/github.com/scip-code/scip/bindings/go/scip#IndexVisitor.ParseStreaming` — documents streaming parsing at document granularity.
- official source: `https://pkg.go.dev/github.com/scip-code/scip/bindings/go/scip#Index` — documents index-level metadata, documents, and external symbols.
- sibling boundary: `epic-planning-1` output CAP-002 — owns the caller-provided index path and shared load failure behavior.

### Description
The traversal layer accepts the successfully loaded SCIP input boundary and presents document-level data in a form query planners can reuse. It preserves index metadata, document identity, document language, document relative paths, document-level symbols, external symbols, and document occurrences. It keeps parsing responsibility centralized so command-specific stories do not define separate SCIP-loading behavior.

## Capability CAP-002 - Preserve source locations and hover metadata

`scip-search` query planners can retrieve source ranges, document paths, symbol roles, and hover documentation for matched SCIP symbols and occurrences.

### References
- goal spec: `README.md#what-is-scip-search` — references and implementations commands answer symbol-location queries from the index.
- official source: `https://raw.githubusercontent.com/scip-code/scip/main/scip.proto` — documents occurrence ranges, 0-based line and character positions, position encoding, symbol roles, override documentation, and enclosing ranges.
- official source: `https://pkg.go.dev/github.com/scip-code/scip/bindings/go/scip#SymbolInformation` — documents symbol documentation, display name, kind, signature documentation, and enclosing symbol metadata.

### Description
The traversal layer carries the source-location and hover-related metadata that query planners need when turning SCIP matches into user-visible results. It distinguishes document identity, occurrence range, enclosing range, symbol role bitsets, symbol kind, display name, documentation, signature documentation, and occurrence override documentation. It does not decide which fields appear in a command response; it ensures the query-specific epics can access the data without reparsing SCIP structures.

### Out of Scope

- Converting SCIP ranges to editor-specific coordinate conventions beyond preserving schema-defined range data for query planners.

## Capability CAP-003 - Expose occurrence and relationship lookup views

`scip-search` query planners can ask traversal views for occurrences and SCIP symbol relationships needed to resolve references, definitions, implementations, and related symbols.

### References
- goal spec: `README.md#what-is-scip-search` — documented queries include references and implementations.
- official source: `https://raw.githubusercontent.com/scip-code/scip/main/scip.proto` — documents `Relationship.is_reference`, `Relationship.is_implementation`, `Relationship.is_type_definition`, `Relationship.is_definition`, and `SymbolRole.Definition`.
- official source: `https://pkg.go.dev/github.com/scip-code/scip/bindings/go/scip#Relationship` — documents relationship getters and relationship canonicalization helpers.
- sibling boundary: `epic-planning-4` scope — owns exact query semantics for `references --symbol` and `implementations --symbol`.

### Description
The traversal layer provides reusable lookup views over occurrences by symbol and relationships by symbol so query planners can resolve semantically connected symbols without scanning raw documents repeatedly. It exposes definition and non-definition occurrence roles, import/read/write/test/generated role bits, and relationship edges for reference and implementation expansion. It does not decide which related symbols a query must include; it supplies the relationship data needed by the query-specific stories.

### Story Documents
| Story Doc | Title | Priority | Notes |
|-----------|-------|----------|-------|
| specs/stories/readme/epic-planning-2/CAP-003-01-occurrence-lookup-view.md | Index occurrences by SCIP symbol for query planners | P1 | Depends on CAP-001 document inventory and CAP-002 source range preservation. |
| specs/stories/readme/epic-planning-2/CAP-003-02-relationship-lookup-view.md | Index SCIP relationships for reference and implementation planners | P1 | Depends on CAP-001 symbol inventory. |

### Depends on:
- Capability CAP-001 - Load SCIP index data for query traversal: relationship and occurrence lookup views depend on loaded documents, symbols, and external symbols.
- Capability CAP-002 - Preserve source locations and hover metadata: occurrence lookup results need source ranges and document paths preserved for downstream query output planning.

### Out of Scope
- The final algorithm for `references --symbol`, `implementations --symbol`, missing-symbol behavior, result grouping, duplicate elimination policy, and JSON response schema.
- Synthesizing relationships not present in SCIP data.
- Cross-index or multi-worktree relationship resolution.

### Assumptions
- **ASM-003-1**: Traversal should surface relationship edges exactly enough for query planners to choose expansion rules — *Why*: the SCIP schema distinguishes reference, implementation, type-definition, and definition relationships, while sibling epics own which edges each command follows — Confidence: HIGH.

## Capability CAP-004 - Provide traversal fixtures for SCIP data coverage

`scip-search` maintainers can validate traversal stories against deterministic SCIP fixtures that cover documents, occurrences, symbols, relationships, ranges, and hover metadata used by downstream query planners.

### References
- goal spec: `README.md#language-support` — traversal reads SCIP output directly using official bindings.
- goal spec: `README.md#scip-symbol-format` — examples show full SCIP symbol strings returned for later references and implementations queries.
- official source: `https://raw.githubusercontent.com/scip-code/scip/main/scip.proto` — defines the schema elements fixtures must exercise.
- sibling boundary: `epic-planning-3` and `epic-planning-4` scopes — own query-specific tests and fixtures for command behavior.

### Description
Traversal fixtures give downstream stories a shared, schema-valid baseline that proves the traversal layer can ingest and expose the SCIP data categories used by all query classes. The fixture set includes at least one document with definitions and references, at least one external symbol or documented symbol metadata case, at least one implementation relationship, and ranges that exercise same-line and multi-line forms. Query-specific epics may add their own fixtures, but they do not need to rebuild the common traversal coverage.

### Story Documents
| Story Doc | Title | Priority | Notes |

| specs/stories/readme/epic-planning-2/CAP-004-01-schema-valid-traversal-fixtures.md | Create schema-valid SCIP traversal fixtures | P1 | Depends on CAP-001 through CAP-003 to know the required data coverage. |
| specs/stories/readme/epic-planning-2/CAP-004-02-fixture-coverage-validation.md | Validate fixture coverage for downstream query planners | P2 | Depends on the fixture set. |

### Depends on:

- Capability CAP-002 - Preserve source locations and hover metadata: fixtures must cover range and hover metadata preservation.
- Capability CAP-003 - Expose occurrence and relationship lookup views: fixtures must cover occurrence and relationship lookup behavior.

### Out of Scope
- Installing or invoking external language indexers in test setup.
- Full CLI command tests for symbols, packages, references, implementations, stdout, stderr, or exit statuses.
- Large real-world repository fixtures, performance benchmarking, golden files for final command JSON schemas, and ctags fallback fixture data.

### Assumptions
- **ASM-004-1**: Synthetic fixtures are acceptable when they are valid SCIP payloads read through the same official binding path as real indexes — *Why*: the traversal concern is schema coverage and deterministic data exposure, while indexer installation belongs outside this epic — Confidence: MEDIUM.
