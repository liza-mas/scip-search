# Epic EP-001: CLI Runtime Contract

Status: draft

## Goal
`scip-search` exposes a predictable one-shot CLI runtime contract for command invocation, shared index loading, JSON stdout, stderr diagnostics, and exit behavior.

## Context
The product is a thin Go binary that loads a pre-built SCIP index, answers one query, prints structured JSON, and exits. This epic bounds the runtime behavior shared by all query commands so sibling epics can focus on SCIP traversal and query-specific result content without redefining process semantics.

## Personas
- **Automation agent**: A coding or planning agent running terminal commands inside concurrent worktrees, needing stable machine-readable output and deterministic failures.
- **CLI developer**: A Go developer implementing and testing `scip-search`, needing a bounded runtime contract that separates CLI behavior from query traversal.

## General information

Applies to: the entire epic scope.

### References
- goal spec: README.md#what-is-scip-search, lines 56-86 - Defines the binary as a one-shot command that loads an index, answers a query, prints structured JSON to stdout, and exits.
- goal spec: README.md#out-of-scope, lines 88-96 - Excludes daemon/watch mode, custom index formats, MCP, UI, and related adjacent capabilities.
- task: epic-planning-1 scope - Limits this epic to CLI entrypoint, command/flag contract, index file loading boundary, stdout/stderr/exit behavior, and shared JSON/error conventions.

### Completion Criteria
All `scip-search` commands share a documented runtime contract: command invocation requires the relevant query command and `--index` flag, valid invocations load the provided SCIP index once, successful commands write structured JSON to stdout, failures write diagnostics to stderr, and every invocation exits with a documented status. Query-specific traversal and result fields remain owned by sibling epics.

### Non-Functional Requirements
- NFR-000-1: The runtime reads pre-built SCIP output directly and performs no compilation, type-checking, daemon startup, watch loop, or incremental update work.
- NFR-000-2: The runtime contract supports concurrent worktrees by accepting an explicit index file path instead of assuming a global or repository-local index location.
- NFR-000-3: Successful output must be suitable for automation: stdout is structured JSON and must not contain progress text, warnings, or diagnostics.

### Related External Components
Summary of all external components referenced by this epic:
- Component C-001 - SCIP index file: The on-disk input selected by the caller with `--index`.
- Component C-002 - SCIP Go bindings: The official parsing boundary used by the runtime to read SCIP output.
- Component C-003 - Calling process environment: The shell, script, or agent that observes stdout, stderr, and exit status.

### Interfaces
Summary of all external interfaces referenced by this epic:
- I-003-001 - CLI process contract (Interface 001 of Component C-003): The command-line invocation, stdout/stderr streams, and process exit status exposed by `scip-search`.

### Out of Scope
- SCIP document, occurrence, symbol, relationship, range, and hover traversal details owned by `epic-planning-2`.
- Query-specific behavior for `symbols` and `packages` owned by `epic-planning-3`.
- Query-specific behavior for `references` and `implementations` owned by `epic-planning-4`.
- Installation, release packaging, version command behavior, and user-facing distribution docs owned by `epic-planning-5`.
- Generating SCIP indexes, installing language indexers, daemon/watch mode, MCP server behavior, UI, graph visualization, semantic similarity, vector storage, and custom index formats.
- Implementing code, choosing a CLI framework, or writing user stories in this epic document.

### Assumptions
- **ASM-000-1**: Runtime planning may define shared process behavior without defining query-specific result schemas. - *Why*: The task scope names shared JSON/error conventions while sibling tasks own query behavior. - Confidence: HIGH
- **ASM-000-2**: A nonzero exit status is sufficient at epic level for failure semantics; exact numeric taxonomy can be specified by story acceptance criteria. - *Why*: The source requires the process to exit and the task asks for error semantics, but no numeric codes are named in the goal spec. - Confidence: MEDIUM

## Capability CAP-001 - Invoke scip-search commands consistently

`scip-search` users can invoke documented query commands with shared flags and get deterministic one-shot process behavior before any query-specific traversal runs.

### References
- goal spec: README.md#what-is-scip-search, lines 58-72 - Lists the binary behavior and command forms.
- task: epic-planning-1 scope - Includes CLI entrypoint, command/flag contract, and process behavior.

### Description
This capability defines the command surface shared by `symbols`, `references`, `implementations`, and `packages`. It bounds how required commands and shared flags are accepted, how invalid invocation shape is rejected, and how the process behaves as a single query command rather than a daemon or interactive tool.

## Capability CAP-002 - Load the caller-selected SCIP index

`scip-search` users can point a command at an explicit SCIP index file and receive either a loaded query context or a clear loading failure before query execution.

### References
- goal spec: README.md#what-is-scip-search, lines 58-65 - Requires loading the index file provided as an argument before answering a query.
- goal spec: README.md#language-support, lines 82-86 - Requires official SCIP bindings and direct SCIP output reads.
- task: epic-planning-1 scope - Includes the index file loading boundary.

### Description
This capability defines the runtime boundary around the `--index` path. It covers accepting a caller-provided index path, distinguishing missing or unreadable index input from successful loading, and handing a loaded SCIP index to query-specific behavior without generating or updating indexes.

### Story Documents
The set of story documents this capability decomposes into. One story document per cohesive unit of work a Story Writer can own.

| Story Doc | Title | Priority | Notes |
|-----------|-------|----------|-------|
| specs/stories/readme/index-path-validation.md | Index path validation failures | P1 | Depends on CAP-001 shared `--index` flag behavior. |
| specs/stories/readme/scip-index-loading-boundary.md | SCIP index loading boundary | P1 | Defines success and failure behavior at the official SCIP parse boundary. |

### Depends on:
- Capability CAP-001 - Invoke scip-search commands consistently: The runtime must define the shared `--index` flag before index loading stories can bind loading behavior to invocation behavior.

### Out of Scope
- Generating indexes with language indexers.
- Watching, caching, incrementally updating, or storing indexes in custom formats.
- Traversing the loaded index for symbols, packages, references, or implementations.
- ctags fallback wrapper behavior.

### Assumptions
- **ASM-002-1**: The runtime treats the index path as caller-owned input and does not search default locations. - *Why*: The goal spec examples always pass `--index`, and concurrent worktrees require multiple explicit index locations. - Confidence: HIGH

### Open Questions
- None.

## Capability CAP-003 - Report machine-readable results and actionable failures

Automation users can rely on stdout for successful structured JSON and stderr plus process status for all runtime failures across `scip-search` commands.

### References
- goal spec: README.md#what-is-scip-search, lines 58-63 - Requires structured JSON to stdout and process exit.
- task: epic-planning-1 scope - Includes JSON stdout plus error and exit semantics.

### Description
This capability defines stream and status conventions shared by every command. It covers successful JSON-only stdout, diagnostic-only stderr on failures, and consistent failure reporting for invalid invocation, missing index input, unreadable index files, and index loading failures.

### Story Documents
The set of story documents this capability decomposes into. One story document per cohesive unit of work a Story Writer can own.

| Story Doc | Title | Priority | Notes |
|-----------|-------|----------|-------|
| specs/stories/readme/stdout-json-success-contract.md | JSON stdout success contract | P1 | Coordinates with sibling query epics without defining query-specific fields. |
| specs/stories/readme/stderr-exit-error-contract.md | stderr and exit status failure contract | P1 | Depends on CAP-001 and CAP-002 failure sources. |
| specs/stories/readme/shared-runtime-error-cases.md | Shared runtime error cases | P2 | Covers invocation and index loading errors only. |

### Depends on:
- Capability CAP-001 - Invoke scip-search commands consistently: Invocation failures are part of the shared error contract.
- Capability CAP-002 - Load the caller-selected SCIP index: Index path and loading failures are part of the shared error contract.

### Out of Scope
- Query-specific result field schemas for symbols, packages, references, or implementations.
- Pretty-printing, progress output, logging configuration, or human-oriented UI output.
- Distribution documentation and install verification commands.

### Assumptions
- **ASM-003-1**: Shared JSON conventions can require JSON-only stdout on success without prescribing each query's result fields. - *Why*: The source requires structured JSON, while sibling query epics own the domain-specific payloads. - Confidence: HIGH

### Open Questions
- None.
