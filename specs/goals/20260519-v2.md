# PRD: scip-search v2 Agent Navigation Loop

Status: draft

## Goal
Make scip-search a deterministic Bash-first code-navigation primitive that lets coding agents find symbols, jump to exact references, and focus-read source with fewer speculative grep/read loops.

## Context
The current tool already demonstrates the core loop: symbol discovery, exact-symbol reference lookup, and narrow source reads. The next iteration should harden that loop, make stale-index behavior explicit, and improve high-signal terminal output without turning scip-search into an IDE or semantic search product.

## General information

Applies to: the v2 command-line experience and query behavior for coding-agent code exploration.

### Non-Functional Requirements
- NFR-000-1: Commands must remain one-shot CLI operations that read a caller-provided index path and exit.
- NFR-000-2: Default text output must remain line-oriented and suitable for Bash pipelines.
- NFR-000-3: JSON output must remain available for machine consumers without replacing the default terminal workflow.
- NFR-000-4: Behavior must be deterministic over the same SCIP index and arguments.

### Related External Components
- Component C-001 - SCIP index files generated by language-specific indexers.
- Component C-002 - Agent shell workflow using `scip-search`, `nl`, and `sed`.

### Out of Scope
- Interactive UI, daemon mode, editor integration, LSP replacement, fuzzy semantic search, automatic index regeneration, and ranking based on hidden heuristics.
- Changing SCIP indexer behavior or requiring a specific language indexer beyond documented examples.
- Solving Liza task graph sequencing inside scip-search.

### Assumptions
- **ASM-000-1**: v2 should optimize for coding agents before human IDE users - *Why*: the README positions the tool as agent-optimized and the observed value is reducing agent grep/read loops - Confidence: HIGH.
- **ASM-000-2**: Exact-symbol lookup should stay central even if name lookup becomes more convenient - *Why*: exact symbols prevent ambiguity and make reference queries reproducible - Confidence: HIGH.

### Open Questions
- **OQ-000-1**: Should stale-index metadata be printed in default text output, only JSON output, or only via an explicit command? - *Impact if unresolved*: stale index warnings may either be too noisy or too easy for agents to miss.

## Feature FT-001 - Preserve the Three-Command Exploration Loop

### References
- README: `README.md:5-49`
- pairing feedback: conversation on 2026-05-19

### Functional Requirements
- FR-001-1: `symbols --name` must return enough information for an agent to choose exact SCIP symbols without opening broad source files.
- FR-001-2: `references --symbol ... --location-only` must return only source locations, one per line, suitable for direct use with `nl` and `sed`.
- FR-001-3: Multi-symbol queries must preserve deterministic output ordering and de-duplicate repeated resolved symbols.

### Acceptance Criteria
- AC-001-1: Given an index containing multiple symbols named `Foo`, when `symbols --index <index> --name Foo` runs, then each result includes path, line, column, exact symbol, match source, and match text.
- AC-001-2: Given an exact symbol, when `references --index <index> --symbol <symbol> --location-only` runs, then stdout contains only `<path>:<line>:<column>` lines and stderr is empty.
- AC-001-3: Given repeated names or symbols resolving to the same exact symbol, when a query runs, then the symbol appears once in deterministic order.

### Out of Scope
- Fuzzy matching, semantic ranking, source-file preview rendering, and automatic `nl`/`sed` execution.

## Feature FT-002 - Make Index Freshness Explicit

### References
- README: `README.md:77-85`
- issue log: `liza-run-issues.md:70-86`
- pairing feedback: conversation on 2026-05-19

### Functional Requirements
- FR-002-1: Documentation must state that SCIP indexes are snapshots and do not reflect source edits made after index generation.
- FR-002-2: JSON output must include index metadata sufficient for callers to identify which index file was queried.
- FR-002-3: Default text output must not silently imply live-source freshness.

### Acceptance Criteria
- AC-002-1: Given a user reads the README usage loop, when they reach the first executable examples, then the stale-index caveat is visible nearby.
- AC-002-2: Given a query runs with `--json`, when output is parsed, then the response identifies the queried index path or equivalent stable index identity.
- AC-002-3: Given an index is older than the current source tree, when scip-search cannot prove freshness, then docs and/or metadata make clear that results are index-derived, not live-source-derived.

### Out of Scope
- Automatically rebuilding indexes, comparing every indexed file against the filesystem, or refusing to run on old indexes.

## Feature FT-003 - Keep Output Boring and Composable

### References
- README: `README.md:163-190`
- pairing feedback: conversation on 2026-05-19

### Functional Requirements
- FR-003-1: Default output must remain grep-style, one result per line.
- FR-003-2: Commands must keep stdout for successful results and stderr for diagnostics.
- FR-003-3: Output modes must be explicit flags rather than environment-sensitive behavior.

### Acceptance Criteria
- AC-003-1: Given a successful default query, when stdout is piped to standard Unix tools, then each result is independently parseable as one line.
- AC-003-2: Given a usage error, when the command exits, then stdout is empty, stderr is non-empty, and the exit status indicates usage failure.
- AC-003-3: Given the same index and arguments, when a query is run twice, then output ordering is identical.

### Out of Scope
- Colorized output by default, progress bars, paging, prompts, or terminal-width-dependent formatting.
